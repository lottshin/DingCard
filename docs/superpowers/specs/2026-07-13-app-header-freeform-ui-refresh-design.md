# A 方案顶栏与自由编辑器 UI 统一设计

日期：2026-07-13  
分支：`feature/freeform-editor`  
状态：用户已确认采用 A 方案，并同意同时美化自由编辑器主体区域。

## 1. 背景

当前应用已经有两套独立工作区：Markdown 卡片和自由编辑。功能边界是正确的，但界面层级存在以下问题：

- 工作区切换器独立悬浮在页面顶部，和两个工作区自己的顶栏没有形成统一导航。
- Markdown 与自由编辑分别维护顶栏、主题、登录和草稿入口，切换工作区时容易出现视觉与状态不一致。
- 顶栏按钮同权重排列，模式切换、编辑设置、保存、草稿、导出和账户操作混在一起。
- 自由编辑器虽然已有页面栏、画布和属性栏，但容器、控件、间距和状态样式不够统一。
- 部分输入框、滑动条、颜色入口和按钮组仍容易呈现浏览器原生感，和产品现有 UI 不一致。
- 页面背景、文字颜色、形状填充和图片填充属于同一套视觉能力，但在属性面板中的分组层级仍需统一。

用户在可交互原型中比较了 A、B、C 三套结构，最终确认 A：两层工作台。

## 2. 目标

本轮交付包含两部分：

1. 将应用顶部改为 A 方案的两层结构：全局导航 + 当前工作区工具栏。
2. 在不改变自由编辑功能和文档模型的前提下，统一页面栏、画布区、属性栏和所有表单控件的视觉与交互层级。

具体目标：

- Logo 与工作区切换成为稳定的全局导航，不再作为悬浮胶囊单独占位。
- 每个工作区只在第二层展示与当前任务相关的工具。
- 全局区域只保留全局行为；保存、草稿和导出继续归属于当前工作区。
- 自由编辑器形成清晰的三栏结构：页面栏、画布、属性栏。
- 所有可见的下拉框、输入框、滑动条、颜色按钮、文件入口和按钮组使用同一套项目 UI。
- 纯色、渐变、透明和图片填充按对象能力统一归入“填充”分组。
- 深色与浅色主题、1440×900 与 1024×768 桌面视口均可正常使用。
- Markdown 内容编辑区、卡片主题和导出效果保持不变。

## 3. 非目标

- 不重做 Markdown 编辑器或卡片预览区域。
- 不改变自由编辑文档结构、草稿格式、撤销重做历史或导出渲染结果。
- 不新增裁剪、图层面板、模板库、快捷命令面板或富文本能力。
- 不进行移动端编辑器重构。本轮最低支持桌面宽度为 1024px。
- 不把原型中的示例画布内容、示例图片或装饰图形加入正式产品。
- 不引入新的大型 UI 组件库或图标库。
- 不为了视觉统一重写现有 `PaintField`、拖拽、吸附或选择框算法。

## 4. 方案选择

### 4.1 采用方案：A 两层工作台

第一层是全局导航，高度 52px：

```text
[叮卡 Logo] [Markdown 卡片 | 自由编辑]                         [主题] [登录/头像]
```

第二层是工作区工具栏，高度 50px，并随工作区切换：

```text
Markdown： [平台] [主题] [字体] [个人资料]              [保存草稿] [草稿] [导出]
自由编辑： [页面尺寸] [文本] [图片] [形状] [线条] [撤销/重做] [保存] [草稿] [全部导出] [导出当前页]
```

两层总高 102px。当前界面由工作区切换胶囊和 60px 工作区顶栏组成，实际占高接近该数值，因此 A 方案不会明显减少可用画布高度。

### 4.2 未采用方案

- B 单层紧凑顶栏：短期占高较少，但功能继续增加后会再次拥挤，1024px 下尤其明显。
- C 左侧模式栏：更接近专业设计软件，但结构改动大，并会进一步压缩横向画布空间。

## 5. 信息架构

### 5.1 全局导航

全局导航只承载跨工作区行为：

- 产品 Logo 与名称。
- Markdown 卡片 / 自由编辑切换。
- 应用深浅色切换。
- 登录按钮或用户头像。

草稿、保存和导出不放入全局导航。它们依赖当前工作区文档状态，放在第二层可以避免在 `AppShell` 中建立跨工作区命令注册或隐式回调。

### 5.2 Markdown 工作区工具栏

保留现有业务能力与排列逻辑，仅进行以下结构调整：

- 移除工作区内部重复的 Logo、主题切换和登录入口。
- 平台、卡片主题、字体和个人资料位于左侧。
- 保存草稿、草稿列表和批量导出位于右侧。
- 导出是唯一强主按钮，使用品牌橙色实心样式。
- 保存与草稿使用次级按钮，避免和导出争夺层级。

Markdown 主体编辑区、预览区、分页和导出交互不变。

### 5.3 自由编辑工作区工具栏

- 页面比例/像素尺寸作为第一个上下文入口。
- 插入能力按“文本、图片、形状、线条”分组。
- 形状的具体类型不再全部铺成顶栏同级按钮；点击“形状”后使用现有风格的轻量菜单展示矩形、圆形、三角形等选项。
- 线条入口同样可在轻量菜单中选择直线或箭头。
- 撤销、重做使用图标按钮并保留 tooltip 与 `aria-label`。
- 保存、草稿、全部导出为次级按钮；导出当前页是唯一强主按钮。
- 导出期间保留现有进度文案和 disabled 行为。

## 6. 自由编辑器主体视觉

### 6.1 页面栏

- 建议宽度 152px，使用较浅的工作区表面色，与画布背景区分。
- 顶部只有“页面”和新增按钮，不增加额外功能。
- 缩略图保持真实页面比例；选中页使用 2px 品牌橙边框和轻阴影。
- 页码与缩略图分离显示，避免页码压住内容。
- 新增、复制、删除继续使用现有业务逻辑；危险操作使用明确的危险色 hover/focus 状态。
- 缩略图列表滚动条使用项目统一样式。

### 6.2 画布区

- 使用中性暖灰背景，让白色与透明页面都能被识别。
- 页面使用克制阴影，不增加纹理、渐变网格或装饰性背景。
- 画布缩放、页面尺寸和当前页信息使用低强调文字，不和编辑内容竞争。
- 选择框、控制点、吸附线继续使用现有交互颜色；仅统一线宽、阴影和深浅色对比度。
- 画布 DOM 和导出 DOM 的职责不变，UI 阴影、控制点和参考线不得进入导出结果。

### 6.3 属性栏

- 建议宽度 248px；每个属性分组使用标题、内容、分隔线组成，不堆叠大量独立卡片。
- 未选中元素时显示页面属性。
- 单选文本、形状、图片、线条时显示对应属性。
- 多选时只显示共同操作与现有对齐/分布能力，不伪造不支持的混合属性编辑。

属性顺序：

1. 对象/页面基本信息。
2. 位置与尺寸。
3. 文字排版（仅文本）。
4. 填充。
5. 描边或线条。
6. 排列、对齐和分布。
7. 删除等危险操作。

### 6.4 填充分组

填充是一个统一能力，不再把颜色、渐变和图片入口拆成互不相关的区域：

- 页面：纯色 / 渐变 / 透明。
- 文本：纯色 / 渐变。
- 形状：纯色 / 渐变 / 图片。
- 线条与箭头：本轮仍为纯色描边，但使用相同颜色入口外观。
- 图片元素：保留图片适配与现有能力，不伪装成颜色填充。

继续复用现有 `PaintField` 与颜色弹层逻辑。可见入口必须是项目样式的色块按钮和模式切换；隐藏的原生 `input[type=color]` 或文件输入只负责调用系统能力，不能直接出现在界面上。

## 7. 视觉系统

### 7.1 尺寸与间距

- 全局导航：52px。
- 工作区工具栏：50px。
- 常规控件高度：32px。
- 图标按钮：32×32px，点击目标通过容器或间距保证不小于 40px。
- 控件圆角：8px；分组圆角：10px。
- 工具栏横向间距：6–8px；属性栏字段纵向间距：10–12px。

### 7.2 颜色与层级

- 品牌主色沿用橙红色系，主按钮、当前页面和关键 focus 状态统一使用同一 token。
- 当前模式不使用黑色实心块，改为浅表面、边框、字重和轻阴影组合。
- 导出是每个工作区唯一实心主按钮。
- 深色模式下次级控件必须有可见背景和边框，不能只依赖低对比度描边。
- 普通正文与背景对比度不低于 4.5:1；非文字控件边界和 focus 状态必须清晰。

### 7.3 图标与反馈

- 使用项目内一致的线性 SVG 图标，不使用 emoji 作为功能图标。
- 图标按钮必须提供 `aria-label` 和 tooltip。
- hover/focus/pressed/disabled/loading 状态齐全。
- hover 只改变颜色、边框、阴影或最多 1px 位移，不改变控件占位尺寸。
- 动画控制在 150–220ms，并尊重 `prefers-reduced-motion`。

## 8. 组件与状态边界

### 8.1 新增或调整的组件

建议组件边界：

```text
src/workspaces/AppShell.tsx
src/workspaces/AppHeader.tsx
src/workspaces/WorkspaceToolbar.tsx
src/freeform/FreeformInsertMenu.tsx
src/freeform/InspectorSection.tsx
```

职责：

- `AppShell`：持有 `workspaceMode`、全局主题和当前用户，渲染全局导航与当前工作区。
- `AppHeader`：纯展示全局导航；接收模式、主题、用户和回调，不读取自由编辑文档。
- `WorkspaceToolbar`：提供左右分区、分隔线和响应式容器，不包含具体业务状态。
- `FreeformInsertMenu`：负责形状/线条二级菜单的显示和键盘操作，最终仍调用现有 `addShape` / `addLine`。
- `InspectorSection`：统一属性组标题、间距、分隔线和折叠语义；第一版默认展开，不新增持久化折叠状态。

组件名称在实施计划中可根据现有代码组织微调，但职责不能重新混在单个超大组件中。

### 8.2 全局状态

`useAppTheme` 与当前登录用户提升到 `AppShell`，通过 props 传给两个工作区：

- 切换工作区后主题图标、登录态和头像必须立即一致。
- 登录弹层的开关状态和登录/退出动作由 `AppShell` 持有；`AppHeader` 与工作区通过明确的 `requestAuth`/账户回调请求打开，不各自创建第二套登录 state。
- 登录/退出弹层只保留一套全局实例。
- 两个工作区继续独立管理自己的文档、草稿列表、保存、导出和工作区弹层。
- 不新增全局事件总线，不使用 `window` 自定义事件同步 React 状态。

### 8.3 工作区挂载策略

本轮保留两个工作区持续挂载、通过 `hidden` 切换的现有策略，以避免切换模式时丢失未保存编辑状态。

全局主题和用户提升后，隐藏工作区收到相同 props，不再保留独立且可能过期的主题/登录 state。

隐藏工作区必须继续使用原生 `hidden` 语义或等价的 `display: none` + `aria-hidden` 方案，确保其顶栏、菜单和表单控件不进入键盘顺序与可访问性树。

## 9. 响应式行为

### 9.1 1280px 及以上

- 所有主要工具显示图标与文字。
- 左侧页面栏 152px，右侧属性栏 248px。
- 导出按钮完整显示文案。

### 9.2 1024–1279px

- 自由编辑插入、撤销和重做按钮可只显示图标，tooltip 与 `aria-label` 保留完整含义。
- 次要分隔间距缩小，但控件高度不变。
- 页面栏可缩到 136px，属性栏可缩到 224px。
- 保存、草稿和当前页导出仍必须可见；全部导出允许缩短文案但不能移除。
- 页面整体不得产生横向滚动条，画布区可以按现有逻辑缩放。

### 9.3 小于 1024px

本轮不承诺完整移动编辑体验。界面可以显示桌面宽度提示，但不得崩溃、遮挡登录/退出入口或产生无限布局抖动。

## 10. 错误与边界行为

- 形状或线条菜单关闭、点击外部或按 Escape 时，不创建元素。
- 图片文件选择取消时保持原文档不变。
- 图片填充从纯色/渐变切换后又取消选图时，保留原填充，继续遵循现有契约。
- 没有选中元素时，属性栏稳定回退到页面属性，不显示空白面板或旧元素属性。
- 多选包含不同类型元素时，不展示无法安全批量修改的属性。
- 无法读取当前用户时回退为未登录状态；不影响本地未保存编辑内容。
- 导出或保存过程中按钮禁用规则、进度文案和失败反馈保持现有行为。
- 深浅色切换不得改变卡片/页面导出的内容主题。

## 11. 测试设计

### 11.1 单元测试

- 全局主题与用户状态提升后，现有纯函数和存储函数继续通过。
- 若新增工具栏响应式或属性分组映射纯函数，为 0、空选择、未知元素类型和多选混合类型补充稳定 fallback 测试。
- 不为纯 CSS 细节编写脆弱的字符串快照测试。

### 11.2 E2E 测试

扩展现有 Playwright 用例：

- 默认进入 Markdown，工作区标签名称与 `aria-selected` 正确。
- 切换自由编辑后，A 方案全局导航保持不变，第二层工具栏替换为自由编辑工具。
- 在一个工作区切换深浅色或登录后，切换到另一个工作区，主题和用户状态一致。
- 自由编辑的文本、图片、形状、直线和箭头仍能从新入口插入。
- 形状和线条菜单支持鼠标、键盘、Escape 和点击外部关闭。
- 页面比例、自定义尺寸、撤销、重做、保存草稿、草稿打开、当前页导出和批量导出继续可用。
- 选中页面、文本、形状、图片、线条和多选时，属性栏显示正确分组。
- 纯色、渐变、透明和图片填充入口仍遵循现有能力矩阵。
- 1440×900 与 1024×768 下无页面横向滚动，主导出按钮和属性栏可见。
- 浅色与深色模式下，输入框、滑动条、颜色入口、菜单和 disabled 状态都可辨认。

### 11.3 回归测试

- Markdown 编辑、分页、卡片预览和 ZIP 导出通过。
- 自由编辑选择、拖拽、缩放、旋转、复制粘贴、对齐、分布和吸附通过。
- 单页 PNG 与批量 ZIP 不包含顶栏、属性栏、选择框、参考线或菜单。
- 导出回归测试直接检查导出根节点不包含应用主题 class/data 属性、顶栏、弹出菜单和编辑辅助层，避免仅凭肉眼判断截图。
- 中文输入法、字体选择、渐变文字和图片填充通过现有风险用例。
- `npm run build`、`npm run test:unit`、完整 Playwright E2E 通过。

## 12. 实施顺序

1. 提升全局主题与用户状态，建立 `AppHeader`，保持工作区业务功能不变。
2. 将 Markdown 与自由编辑顶栏改为共享的第二层工具栏骨架。
3. 重组自由编辑插入入口，确保旧 handler 和测试契约有明确迁移。
4. 美化页面栏、画布区和属性栏容器。
5. 用统一 token 重做可见输入框、滑动条、颜色入口、菜单和按钮状态。
6. 整理属性栏分组，但不改变文档数据和渲染函数。
7. 补充 1440×900、1024×768、light/dark 和跨工作区状态 E2E。
8. 完整回归并检查导出视觉不受 UI CSS 污染。

## 14. 验收标准

- 顶部采用 A 方案两层结构，不再出现独立悬浮的工作区切换胶囊。
- 全局导航只包含 Logo、工作区切换、应用主题和账户。
- Markdown 内容区功能与视觉主体不变，仅接入新顶栏。
- 自由编辑页面栏、画布和属性栏达到原型所示的统一层级。
- 形状和线条不再以大量同级按钮挤满顶栏。
- 纯色、渐变、透明和图片填充在属性栏中按“填充”能力统一组织。
- 可见控件不再呈现明显的浏览器原生 select/color/range/file 样式。
- 切换工作区后主题与登录状态一致，未保存文档不丢失。
- 1024×768 和 1440×900 下主操作可见且页面无横向滚动。
- light/dark、登录/未登录、空白页/复杂页、单选/多选、系统字体/网络字体组合均完成验证。
- Markdown 与自由编辑的保存、草稿和导出能力无回归。
- 构建、单元测试和完整 E2E 全部通过。
