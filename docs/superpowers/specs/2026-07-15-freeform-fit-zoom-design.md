# 自由编辑画布适应视口缩放设计

- 日期：2026-07-15
- 分支：`feature/freeform-editor`
- 状态：用户已批准方案 A（`100% = 完整适应当前视口`），主线程自检修订完成

## 1. 背景与根因

自由编辑当前把工具栏百分比直接当作原始页面像素的 CSS 缩放比例。`previewScale` 固定从 `0.5` 开始，因此 1080×1920 页面在 50% 下仍有 960px 高。

实测结果：

- 1440×900 桌面视口下，默认 3:4 页面在 50% 时已经纵向溢出约 38px。
- 同一视口切换到 9:16、1080×1920 后，画布显示高度为 960px，舞台纵向溢出约 278px。
- 1024×768 下可用空间更小，固定 50% 的溢出更明显。

因此，滚动条不是普通 `overflow` 样式错误，也不是导出 DOM 架构问题，而是“百分比代表原始像素比例”的缩放语义与产品预期不一致。

## 2. 目标

- 自由编辑默认显示 100%，且当前页面完整位于画布视口内。
- 100% 始终表示“完整适应当前舞台”，不表示原始像素的 1:1 显示。
- 50% 表示适应尺寸的一半；110% 表示适应尺寸的 1.1 倍。
- 浏览器尺寸、左右栏宽度、页面比例或自定义像素尺寸变化后，100% 自动重新适应。
- 预览缩放不得改变页面文档坐标、草稿格式、撤销历史或导出像素。
- 拖拽、框选和元素缩放继续准确使用屏幕尺寸与页面坐标之间的真实换算比例。

## 3. 非目标

- 本轮不增加“原始像素 1:1”或“实际尺寸”模式。
- 不修改 Markdown 卡片工作区的缩放语义。
- 不修改自由编辑文档结构、草稿持久化格式或导出管线。
- 不改变吸附阈值、选择模型、元素变形规则或页面尺寸预设。
- 不以隐藏滚动条的方式掩盖真实溢出。

## 4. 采用的缩放模型

缩放拆为三个不同概念：

```text
视口适应比例 fitScale
× 用户显示比例 zoomPercent / 100
= 实际渲染比例 renderScale
```

### 4.1 视口适应比例

`fitScale` 根据舞台可用内容区域与当前页面像素尺寸计算：

```text
fitScale = min(
  stageContentWidth / slideWidth,
  stageContentHeight / slideHeight
)
```

舞台可用宽高必须扣除 `.freeform-stage-scroll` 的实际左右、上下 padding，不能写死为 32px，因为紧凑桌面布局会使用不同 padding。

计算函数使用稳定契约：

```ts
calculateFitScale(
  stageContentWidth: number,
  stageContentHeight: number,
  slideWidth: number,
  slideHeight: number,
): number | null
```

- 舞台或页面任一宽高不是有限正数时固定返回 `null`，不得混用 `undefined`、`0`、`NaN` 或 `Infinity`。
- `fitScale` 状态类型为 `number | null`，初始值为 `null`。
- 尚未得到第一个有效值时，不渲染可交互的 stage box/artboard，并禁用依赖 artboard 的导出操作，避免短暂显示原始大尺寸。
- 得到过有效值后，后续一次测量返回 `null` 时保留最近一次有效比例。
- 初次渲染、工作区隐藏或尚未完成布局时可能测得 0×0；不得把比例写成 0、`NaN` 或 `Infinity`。
- 最终比例应保留足够精度，并对小于 1px 的浮点溢出留出容差，避免 100% 因舍入产生滚动条。
- 不对 `fitScale` 设置 20% 之类的下限；超大自定义页面仍必须能够完整适应。

### 4.2 用户显示比例

`zoomPercent` 是用户在界面中看到和控制的相对比例：

- 初始值：100。
- 中间百分比按钮：恢复 100%，即重新完整适应。
- 减号/加号：每次减少或增加 10 个百分点。
- 范围：10%–400%。该范围只约束用户相对缩放，不约束 `fitScale`。
- 页面切换、比例切换和浏览器尺寸变化时保留当前 `zoomPercent`，只重新计算 `fitScale`。
- 缩放属于当前工作区的临时视图状态，不写入文档、撤销栈或草稿。

采用 10%–400% 是因为原来的 20%–120% 上限基于绝对像素缩放；改成相对适应后，120% 只比完整显示大 20%，不足以承担细节查看。此调整不增加新的 UI 控件。

### 4.3 实际渲染比例

`renderScale` 的类型为 `number | null`；只有 `fitScale` 有效时才计算。所有真实布局和坐标换算只能使用有效的 `renderScale`：

- `.freeform-stage-box` 的显示宽高。
- `.freeform-artboard` 的 CSS `transform: scale(...)`。
- 元素拖拽的屏幕位移转页面位移。
- 框选起止点的屏幕坐标转页面坐标。
- 元素缩放手柄的屏幕位移转页面尺寸。

`zoomPercent` 不能直接进入上述计算。这样才能避免界面显示 100% 时，拖拽误按原始 1:1 比例换算。

自由编辑中的旧 `previewScale` 状态应被 `fitScale`、`zoomPercent` 和 `renderScale` 取代，并在该工作区内 grep 清零。Markdown 工作区拥有独立的同名 `previewScale`，不在本轮改名或修改，禁止不分作用域地全局替换。

## 5. 测量与生命周期

为 `.freeform-stage-scroll` 增加 DOM ref，并通过 `ResizeObserver` 观察舞台尺寸。

重新测量的触发条件：

- 自由编辑工作区从隐藏变为激活。
- 舞台容器尺寸变化，包括浏览器缩放、窗口变化和左右栏响应式宽度变化。
- 当前页面的宽度或高度变化。
- 切换到另一种页面比例或自定义尺寸。

自由编辑工作区非激活时由 `[hidden]` 变成 `display: none`，因此仅注册 `ResizeObserver` 不足以保证重新进入时得到有效尺寸。激活后使用 `useLayoutEffect` 在浏览器绘制前主动测量；如果仍测得 0×0，则保留 `null` 或最近有效值，并在后续观察回调中重试。这样首次进入、页面尺寸切换和重新进入工作区时都不会先闪现错误尺寸。观察器和延迟测量都必须在组件清理时释放。

项目支持的现代桌面 Chrome 原生提供 `ResizeObserver`，本轮不声明旧浏览器兼容 fallback，避免提供行为不完整的降级路径。初次测量、页面尺寸依赖和工作区重新激活仍统一调用同一个测量函数。

滚动条出现或消失可能让容器可用尺寸发生一次小幅变化。状态更新应使用 epsilon 去重，允许观察器完成稳定的第二次测量，不得形成来回更新循环。

## 6. 布局与滚动行为

- `zoomPercent <= 100` 时，页面完整可见，舞台自身不应出现水平或垂直滚动。
- 100% 时页面在扣除舞台 padding 后尽量占满可用区域，并保持原比例。
- `zoomPercent > 100` 时，受限方向允许产生舞台内部滚动。
- 放大后画布四边都必须可以通过滚动到达，不能因居中布局造成顶部或左侧不可达。
- 页面较小时仍允许 100% 放大适应舞台；自由编辑内容是 DOM/矢量布局，预览放大不改变导出质量。
- 舞台之外的应用根节点、页面栏和属性栏不得因画布放大产生文档级横向滚动。

## 7. 控件与可访问性

现有缩放控件位置和视觉样式保持不变，只调整语义与状态：

- 减号按钮的可访问名称保持“缩小画布”。
- 加号按钮的可访问名称保持“放大画布”。
- 百分比按钮显示整数百分比，并提供“适应画布（恢复 100%）”的 title 或等价可访问说明。
- 达到 10% 或 400% 时对应按钮禁用，disabled、hover、focus 状态继续沿用项目 UI。
- 点击目标保持至少 44×44px。

## 8. 导出隔离

导出继续使用页面原始宽高：

- `renderSlideBlob` 的 `width`、`height` 取当前页面逻辑像素尺寸。
- 导出克隆样式继续强制 `transform: none`。
- `fitScale`、`zoomPercent` 和 `renderScale` 不写入导出 DOM 的作品样式。
- 1080×1920 页面无论预览为 50%、100% 还是 400%，导出仍为 1080×1920。

## 9. 测试设计

### 9.1 单元测试

为适应比例计算建立纯函数测试：

- 竖版、横版和正方形页面选择正确的限制方向。
- 超大自定义页面可得到低于 20% 的有效实际比例。
- 0、负数、`NaN`、`Infinity` 等边界输入稳定返回 `null`。
- `renderScale = fitScale × zoomPercent / 100`。

动态 CSS padding 的读取属于 DOM 测量职责，不由纯数学函数伪造证明，改由端到端测试读取 computed style 后验证。

### 9.2 端到端测试

至少覆盖：

- 1440×900 和 1024×768 两种桌面视口。
- 9:16、1:1、16:9、最小合法 128×128 以及最大合法 4096×4096 页面。
- 首次进入自由编辑时，在第一个有效测量完成前不出现可交互的原始尺寸画布；测量完成后再启用依赖 artboard 的操作。
- 在 1440×900 的 32px padding 与 1024×768 的 24px padding 下，实际画布边界均按 computed style 后的内容区适应，禁止在实现或测试中写死 padding。
- 默认 100% 时，页面完整可见，舞台横纵溢出均不超过 1px。
- 50% 的页面边界小于 100%，且仍无舞台滚动。
- 110% 至少在一个受限方向出现舞台内部滚动，页面四边可达。
- 浏览器尺寸变化后，100% 自动重新适应。
- 在非 100%（例如 150%）下切换页面尺寸或调整浏览器尺寸，百分比保持 150%，实际比例按新视口重新计算。
- 切换到 Markdown 再返回自由编辑后重新得到有效比例，并保留离开前的 `zoomPercent`。
- 点击中间百分比按钮恢复 100%；达到 10%/400% 时对应按钮禁用，且不能越界。
- 拖拽、框选、元素缩放在动态 `renderScale` 下坐标准确。
- 现有依赖固定 `0.5` 比例的测试改为读取实际画布边界推导比例。
- 导出图片保持原始像素尺寸，且不同预览百分比下导出结果一致。

### 9.3 全量回归

运行 unit、build 和完整 Playwright E2E，重点关注自由编辑选择、拖拽、缩放、吸附、页面切换、深浅主题和导出。

## 10. 兼容性、文档与版本

- 本次不改变持久化格式，无草稿迁移需求。
- “100% 适应视口”修复同时把可用缩放范围扩展到 10%–400%，按新功能执行 minor bump：版本从 0.7.0 升为 0.8.0。
- 版本号需 grep 全项目；真实应用版本只同步 `package.json`、`package-lock.json` 顶层和锁文件根包三个位置，不能把依赖包版本误判为应用版本。
- 若项目用户文档描述缩放百分比，应同步改为“100% 表示适应当前画布视口”。
- 不新增 API、CLI、配置、错误码或状态码。

## 11. 验收标准

在支持的桌面视口中打开自由编辑，任意常用比例和合法自定义尺寸均默认以 100% 完整显示；50% 不会比 100% 更容易产生滚动；超过 100% 后仅中央舞台按需滚动。整个过程中编辑坐标和导出尺寸保持准确，Markdown 工作区行为不变，自动化测试全部通过。
