# 本地发布收尾设计

## 背景

`0.10.1` 已包含 Markdown 卡片与自由画布两个工作区，自由画布图层树功能已经合并到本地 `master`。当前代码具备完整的单元、后端、浏览器与验收测试，但仓库入口文档、版本变更记录和持续集成仍不完整，`docs/backend-plan.md` 中还有若干已经落地却未勾选的安全项。

本任务只补齐本地仓库的发布准备，不连接远端，不推送，不创建发布或 Git 标签，也不决定开源 LICENSE。

## 目标

1. 新用户能仅凭根目录 README 理解产品模式、安装依赖、启动本地版、运行测试和使用 Docker Compose。
2. `0.10.1` 的主要功能、兼容性边界和修复在 CHANGELOG 中可追溯。
3. 仓库内提供可直接启用的 CI 工作流，覆盖前端单元测试、后端测试、生产构建和完整 E2E。
4. 后端安全清单只勾选同时具备实现/配置证据和自动化测试/冒烟证据的项目；依赖真实域名、主机权限或缺少自动化证据的项目继续保持未完成。
5. 在当前环境验证 Docker Compose 配置和生产构建路径；不能执行的外部环境步骤必须如实记录。

## 非目标

- 不做包体积优化，保留当前约 `1.11 MB`、gzip 约 `369 KB` 的入口包。
- 不增加产品功能，不修改编辑、保存、导出或认证行为。
- 不创建远端仓库，不推送分支，不创建 Git 标签或 Release。
- 不新增 LICENSE；在决定公开方式和许可证后单独处理。
- 不修改或忽略用户现有的未跟踪 `.claude/` 目录。

## 文档设计

### README

新增根目录 `README.md`，采用面向使用者与维护者的单一入口结构：

1. 产品简介与两种工作区的用途。
2. 环境要求：Node.js 20+、npm，以及运行 E2E 时所需的 Chrome。
3. 本地模式快速启动：根目录安装、开发服务器、生产构建与预览。通用 npm 命令使用跨平台写法；复制文件、设置环境变量等有 Shell 差异的步骤分别提供 PowerShell 与 POSIX Shell 示例。
4. 可选后端：安装 `server` 依赖、复制环境变量模板、启动服务，并说明 `VITE_API_BASE` 决定 LocalStore/RemoteStore，切换不会自动迁移数据。
5. Docker Compose：复制根 `.env.example`、设置强随机 `JWT_SECRET`、构建启动及健康检查入口。
6. 测试矩阵：逐条列出现有 npm scripts 的覆盖范围，不引入新的 CLI 契约。
7. 数据与部署提醒：本地浏览器数据、服务端数据目录、HTTPS 与备份要求，并链接 `docs/backend-plan.md` 和 `docs/freeform-editor.md`。

README 不复制整份部署文档，详细配置继续由 `docs/backend-plan.md` 维护，避免两处形成不一致的第二真相源。

### CHANGELOG

新增根目录 `CHANGELOG.md`，使用简化的 Keep a Changelog 结构：

- `Unreleased` 保留为空的后续入口。
- `0.10.1 - 2026-07-20` 的事实来源限定为 `docs/freeform-editor.md`、`docs/backend-plan.md`、当前版本文件和本次版本相关提交，不凭计划文档中未完成的设想补写功能。
- 最低条目集合包括：嵌套场景与图层树、分组/取消分组、锁定/隐藏、v1/v2 草稿迁移到 v3、远程图片租约与所有权保护、嵌套内容保存/导出、数字输入与线条属性面板修复。
- 兼容性边界至少注明：LocalStore 与 RemoteStore 不自动迁移数据、服务端版本仍为 `0.2.0`、混合尺寸导出会先显示确认提示。
- 只记录用户可感知变化和重要兼容边界，不复制提交日志。

## CI 设计

新增 `.github/workflows/ci.yml`。虽然本轮不配置远端，该文件应在将来首次推送后即可工作：

- 触发：`push` 与 `pull_request`。
- 权限：工作流仅授予 `contents: read`。
- 运行环境：`ubuntu-latest`、Node.js 20、npm 缓存；缓存依赖路径同时包含根 `package-lock.json` 和 `server/package-lock.json`。
- 安装：根目录与 `server` 分别执行 `npm ci`。
- 快速验证：前端单元测试、后端测试、生产构建。
- 浏览器验证：安装 Playwright 所需的 Chrome 与系统依赖，运行现有 `npm run test:e2e`，超时上限 15 分钟。
- 失败诊断：浏览器安装失败只依赖 CI 步骤日志；测试失败时仅当 `test-results/**/*` 实际存在才上传产物，缺少产物不得让诊断步骤制造第二个失败。
- 并发：使用工作流名、事件类型和 PR 编号或完整 ref 组成分组；同一事件/同一分支的新运行取消旧运行，`push` 与 `pull_request` 不互相取消。

CI 直接调用现有脚本，不新增 `test:ci` 等重复入口。这样本地与 CI 的函数契约保持一致。

## Docker 与发布验证

本地验证包括：

1. `npm run build` 验证 TypeScript 与 Vite 生产构建。
2. 先检查 Docker CLI 与 Compose 插件。任一缺失时，将 Compose 配置验证和容器冒烟明确记为“未执行”，不能记为失败或通过。
3. CLI 可用时，以测试专用 `JWT_SECRET` 运行 `docker compose config`；退出码必须为 0，且展开结果包含 `server`、`web` 两个服务，才算配置验证通过。
4. Docker 引擎可用时，使用独立 Compose 项目名构建并启动服务；在 60 秒内轮询首页得到 HTTP 200，并从 `/api/health` 得到 HTTP 200 与 `{ "ok": true }`，才算容器冒烟通过。
5. 无论冒烟通过或失败，都关闭该独立 Compose 项目并清理本轮创建的容器、网络与卷；清理目标不得复用用户已有 Compose 项目名。引擎不可用时只报告配置验证结果，不声称容器运行通过。
6. 运行前端单元、后端和完整 E2E 套件，确保文档/CI改动未破坏现有流程。

## 兼容与错误处理

- 文档中的命令必须与 `package.json`、`server/package.json` 和 Compose 文件完全一致。
- CI 使用的 Node 主版本必须满足前后端声明，且不能依赖开发机的全局包。
- 浏览器安装失败时依靠步骤日志定位；测试失败时保留实际生成的 Playwright 结果。不通过隐藏错误或允许失败来换取绿色状态。
- 所有示例密钥仅使用明确标注的占位值；任何真实密钥、`.env`、数据库和上传文件都不得进入仓库。

## 验收标准

- README 中的脚本、端口、环境变量和数据模式经仓库级搜索确认无命名冲突。
- CHANGELOG 版本与根 `package.json` 的 `0.10.1` 一致；本任务不改变前后端版本号，因为没有新增产品功能或修复运行时代码。
- CI YAML 可解析，步骤调用现有脚本，Chrome 安装与 Playwright `channel: 'chrome'` 契约一致。
- 后端清单的每个已勾选项目均能同时指向实现/配置证据和自动化测试/冒烟证据；HTTPS、主机权限和缺少直接自动化覆盖的项目仍未勾选。
- Docker CLI/Compose 缺失时结果明确为未执行；可用时 `docker compose config` 满足服务与退出码判据，Docker 引擎可用时容器冒烟还满足首页/API 健康检查与独立项目清理判据。
- 前端单元测试、后端测试、生产构建和完整 E2E 均通过。
- 根工作树的 `.claude/` 保持不变；任务分支只包含发布收尾相关文件。
